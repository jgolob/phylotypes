Phylotyping from JPLACE with LWR-Pregrouping and PD Clustering

I provide a scalable, checkpointed phylotyping script that takes a JPLACE file (e.g. from pplacer or epa-ng) and clusters placed sequence variants (SVs) into phylotypes using:
	1.	Like-weight ratio (LWR)-based pregrouping to avoid all-vs-all distances,
	2.	Tree-aware merging of provisional groups using LCAs and phylogenetic distance, and
	3.	Within-group clustering using a precomputed phylogenetic distance (PD) matrix.

The implementation is in:

src/phylotypes/light_parallelized_checkpointed_phylotyping.py

It is designed for large JPLACE files (10⁴–10⁵ SVs) and supports checkpointing, restart, and optional parallelism in the refine stage.

⸻

1. Inputs and Outputs

Input
	•	A JPLACE file (--jplace) generated by pplacer or epa-ng, with at minimum:
	•	fields including:
	•	edge_num
	•	like_weight_ratio
	•	distal_length
	•	placements: list of placement records, each with:
	•	p: list of placement entries (one per candidate edge)
	•	nm and/or n: SV names associated with that placement
	•	tree: Newick tree string where SEPP-like edge annotations have the form:
	•	name:length[edgeid]

The script normalizes this tree to a form where edge IDs are used as node names, so they can be indexed consistently.

Output
	•	A single phylotypes long-format CSV (--out) with columns:

phylotype,sv
pt__00001,SV_1
pt__00001,SV_27
pt__00002,SV_10
...


	•	Phylotype IDs are of the form pt__00001, pt__00002, …, assigned by decreasing phylotype size (largest phylotype gets pt__00001).

⸻

2. High-Level Algorithm

Given a JPLACE file, the algorithm proceeds in three logical stages:
	1.	Parse tree and placements
	2.	Pregroup SVs by LWR overlap and tree proximity
	3.	Refine each coarse group into phylotypes via PD-based clustering

2.1 Parsing the tree and placements
	•	The tree is normalized so that SEPP-style annotated edges like:

Name:0.0123[123]

become:

{123}:0.0123[123]

i.e., the edge ID 123 becomes the node “name”. This lets us map placements to tree nodes by edge_num.

	•	The tree is read into a skbio.TreeNode via Bio.Phylo → Newick → TreeNode.read.
	•	placements are converted into a mapping:

self.sv_nodes[sv_name] = {
    edge_id: placement_vector  # includes LWR and distal_length
    ...
}

Each SV gets a dict from edge_num → [edge_num, ... like_weight_ratio, ..., distal_length].

⸻

3. Distance Between SVs

The script defines a custom SV–SV distance using the placement profiles on the tree.

For an SV, its “profile” is:

{edge_id: (LWR, distal_length_or_zero)}

depending on whether --no-distal-length is set.

Given two SVs, sv1 and sv2:
	1.	Compute:
	•	sv1_p = profile(sv1)
	•	sv2_p = profile(sv2)
	•	Total LWR per SV: sv1_lwr_total, sv2_lwr_total (with tiny epsilon if zero)
	2.	Split edges into:
	•	Overlap: edges present in both profiles
overlap_nodes = set(sv1_p) ∩ set(sv2_p)
	•	Distant: edges present in at least one SV but not both
distant_nodes = (set(sv1_p) ∪ set(sv2_p)) − overlap_nodes
	3.	Overlap contribution
For each overlapping edge n:

contribution =
   distal_length_sv1(n) * (LWR_sv1(n) / sv1_lwr_total)
 + distal_length_sv2(n) * (LWR_sv2(n) / sv2_lwr_total)

Sum this over all overlapping edges.

	4.	Distant contribution
If there are distant edges:
	•	Compute svp_lca = LCA of all distant edges on the tree.
	•	For each distant edge nid in sv1:

contribution_sv1 =
  (distal_length_sv1(nid) + distance(tree, nid, LCA)) * (LWR_sv1(nid) / sv1_lwr_total)


	•	Similarly for distant edges in sv2.

The final SV–SV distance is the sum of the overlap and distant contributions.

This is cached in two ways:
	•	In-memory cache: a dict keyed by (min(sv1, sv2), max(sv1, sv2), no_dl_flag).
	•	Optional SQLite cache (if --workdir is set): persistent distances(sv1, sv2, no_dl, d) table with WAL enabled.

⸻

4. Stage 1: Pregrouping by LWR Overlap and LCA Distance

Pregrouping reduces the “big whale” N×N SV distance problem to smaller chunks.

4.1 Seed-based LWR grouping (Taichi kernel)

Function: __pregroup_by_LWR__(pd_threshold, lwr_overlap)
	1.	Sort SVs by number of placements (len(self.sv_nodes[sv])), fewest first.
	2.	Maintain a list sv_to_group of SVs not yet assigned to any group.
	3.	Iteratively:
	•	Pop one seed SV → seed_sv.
	•	Initialize group_svs = {seed_sv}.
	•	Define group_node_ids = set(self.sv_nodes[seed_sv].keys()).
	•	For all remaining SVs in sv_to_group, build a matrix:

g_overlap_mat[sv_index, edge_index] = LWR of that SV on that edge (0 if absent)


	•	Use a Taichi kernel LWR_Filter to compute, for each SV, the row sum of LWR across group_node_ids and flag it as:
	•	“In the group” if sum ≥ lwr_overlap,
	•	“Out” otherwise.
	•	Add all flagged SVs to group_svs and remove them from sv_to_group.

	4.	Repeat until sv_to_group is empty.

The result is a list of provisional LWR groups: sv_groups.

4.2 LCA-based merging of provisional groups

Because some provisional groups may still be too granular or too close on the tree, we merge them using LCAs:
	1.	For each provisional group:
	•	Gather all edges touched by its SVs.
	•	Compute the LCA of those edges on the tree.
	2.	Build a group-by-group LCA distance matrix:

g_lca_mat[i, j] = PD between LCA_i and LCA_j


	3.	Run agglomerative clustering on this matrix using _agglom_precomputed(pd_threshold):
	•	Metric: 'precomputed' (or legacy 'affinity' fallback)
	•	Linkage: 'average'
	•	distance_threshold = pd_threshold
	4.	Groups whose LCAs are closer than threshold_pd are merged. The final merged groups are stored as:

self.sv_groups  # coarse groups after LWR + LCA merge



These coarse groups are what the refine stage will further cluster using full SV–SV distances.

⸻

5. Stage 2: Refinement into Phylotypes via PD Clustering

Function: group_features(lwr_overlap, pd_threshold, no_dl, n_jobs)

This stage takes self.sv_groups and, within each group, performs standard hierarchical clustering with the custom PD metric.

5.1 Checkpointing setup
	•	If --workdir is set, the script uses:
	•	pregroups.csv — mapping from coarse group ID to SVs
	•	phylogroups_partial.csv or phylogroups_partial.part-XX.csv — partial phylotype assignments
	•	_last_completed_group.txt or .done markers — to support resume.
	•	If --resume is given:
	•	For pregroups: if pregroups.csv exists, it’s not re-written.
	•	For refine: processing resumes after the last completed group (serial) or shard (parallel).

5.2 Serial refine (default, n_jobs = 1)

For each coarse group g_i in self.sv_groups:
	1.	If the group has a single SV:
	•	It becomes a single phylotype {sv}.
	2.	Otherwise:
	•	Sort the SV list g_sv.
	•	Build an n×n SV–SV PD matrix for this group using sv_pair_distance(...).
	•	Run:

AgglomerativeClustering(
    n_clusters=None,
    distance_threshold=pd_threshold,
    metric='precomputed',
    linkage='average'
)


	•	Each cluster label corresponds to a phylotype (set of SVs) for this coarse group.

	3.	All phylotypes are appended to self.phylogroups as sets of SV names.

If --workdir is set, the script streams checkpoints (temporary phylotype labels with SVs) to phylogroups_partial.csv and updates _last_completed_group.txt every 100 groups.

5.3 Parallel refine (n_jobs > 1 with --workdir)

If n_jobs > 1 and --workdir is provided:
	1.	Coarse groups are partitioned into shards based on group index modulo n_jobs.
	2.	Each shard runs in a separate multiprocessing.Process with its own:
	•	Partial file: phylogroups_partial.part-XX.csv
	•	Done marker:  phylogroups_partial.part-XX.done
	3.	Each shard process:
	•	Iterates over its assigned group indices.
	•	For each group, performs the same within-group PD clustering as in serial mode.
	•	Writes rows of the form:

phylotype,sv,group_index
pt__00012_001,SV_A,12
pt__00012_001,SV_B,12
pt__00012_002,SV_C,12
...


	4.	After all shards finish, the main process:
	•	Reads all partial shard CSVs.
	•	Reconstructs phylotypes per group index, ignoring the temporary phylotype IDs.
	•	Populates self.phylogroups as sets of SVs (just like serial mode).

If n_jobs > 1 but no --workdir is provided, the script falls back to serial mode and logs a warning.

⸻

6. Final Output: Phylotype IDs and Ordering

Phylotypes are stored as self.phylogroups ≈ list of set(SV).

to_long() then:
	1.	Computes (phylotype_index, phylotype_size) for each phylotype.
	2.	Sorts by descending size.
	3.	Emits (phylotype_label, sv) pairs where phylotype_label = "pt__%05d" % (rank + 1).

So:
	•	ID assignment is deterministic given the same input and parameters.
	•	Temporary phylotype labels used for checkpointing (pt__%05d_%03d) are not visible in the final output.

7. Command-Line Usage

Basic invocation:

python light_parallelized_checkpointed_phylotyping.py \
  --jplace placements.jplace \
  --out phylotypes_long.csv

Recommended parameters for typical runs:

python light_parallelized_checkpointed_phylotyping.py \
  --jplace placements.jplace \
  --out phylotypes_long.csv \
  --lwr-overlap 0.01 \
  --threshold_pd 0.1 \
  --workdir /path/to/workdir \
  --n-jobs 8 \
  --log-file phylotyping.log \
  --log-level INFO

Key flags:
	•	--lwr-overlap
	•	Minimum total LWR overlap (across seed edges) to join a provisional group.
	•	Higher → stricter pregrouping (more, smaller provisional groups).
	•	--threshold_pd
	•	Used twice:
	1.	As LCA distance threshold when merging provisional groups.
	2.	As SV–SV distance threshold for within-group hierarchical clustering.
	•	Smaller → more phylotypes (tighter clusters), larger → fewer phylotypes.
	•	--no-distal-length
	•	Ignores distal length in the SV–SV distance.
	•	Useful if distal lengths are noisy or you want a placement-based, branch-length-free metric.
	•	--workdir
	•	Enables checkpointing and SQLite distance cache.
	•	Required for parallel refine (n_jobs > 1).
	•	--resume
	•	Resume from --workdir checkpoints if present.

9. Notes, Assumptions, and Limitations
	•	Field assumptions: the JPLACE file must include edge_num, like_weight_ratio, distal_length in fields.
	•	Memory scaling:
	•	The algorithm avoids building one global N×N matrix by pregrouping and clustering within groups.
	•	Large coarse groups still require all-vs-all PD within that group; highly uneven groups may need parameter tuning.
	•	Taichi: required for the LWR_Filter kernel. If Taichi is problematic in a given environment, the pregrouping step can be reimplemented with pure NumPy.
	•	SQLite cache:
	•	Only used when --workdir is set.
	•	Meant as a performance optimization; failure of the cache does not invalidate the algorithm, it just slows it down.
